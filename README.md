## 📌 О проекте DigGram

**DigGram** — это альтернативная серверная и клиентская архитектура, использующая мощный и безопасный движок **TDLib** (Telegram Database Library). 

Мы проектируем проект так, чтобы **любой желающий мог форкнуть этот репозиторий** и развернуть изолированную, суверенную копию мессенджера за несколько минут. Вся тяжелая логика, базы данных и трафик инкапсулируются внутри вашего личного сервера.

## 🖥️ Системные требования к серверу (Server Tiers)

Перед тем как сделать форк, выберите конфигурацию вашего VPS/VDS в зависимости от планируемого количества пользователей:

| Уровень | Назначение / Масштаб | Онлайн-пользователи | Процессор (CPU) | Оперативная память (RAM) | Диск (NVMe/SSD) | Сетевой канал |
| :---: | :--- | :---: | :--- | :---: | :--- | :--- |
| **A** | 🟢 **Лайтовый** (Для себя и пары кентов) | 1 – 5 | 1 Ядро (≥ 2.0 GHz) | 1 GB | 15 – 20 GB | 100 Mbps |
| **B** | 🟢 **Семейный** (Закрытая конфа) | 5 – 30 | 1 – 2 Ядра | 2 GB | 30 – 50 GB | 100 – 300 Mbps |
| **C** | 🟢 **Клубный** (Локальное комьюнити) | 30 – 200 | 2 Ядра (High-Perf) | 4 GB | 80 – 120 GB | 500 Mbps |
| **D** | 🟡 **Локальный паблик** (Приватный чат) | 200 – 1 000 | 4 Ядра | 8 GB | 200 – 300 GB | 1 Gbps (Безлимит) |
| **E** | 🟡 **Медиум** (Полноценный проект) | 1 000 – 5 000 | 8 Ядер | 16 GB | 500 GB+ | 1 Gbps (Выделенный) |
| **F** | 🔴 **Корпорация** (Большой канал) | 5 000 – 10 000 | 16 Ядер (Dedicated) | 32 GB | 1 TB+ (RAID-1) | 1 – 2 Gbps |
| **G** | 🔴 **Громадная сеть** (Масштаб платформы) | 10 000 – 50 000+ | 32+ Ядра (Кластер) | 64 – 128 GB | NVMe Массив + S3 | 10 Gbps + DDoS-защита |

> ⚠️ **Важно для форкеров:** Начиная с **Уровня D**, настоятельно рекомендуется выносить базу данных PostgreSQL/MySQL в отдельный изолированный контейнер и настраивать агрессивное кэширование в ОЗУ.

## 🌐 Спецификация хостингов и баз данных (Infrastructure Matrix)

Анализ платформ для развертывания серверной части `DigGram`, запуска плагинов и хранения баз данных (DB):

| Сервис / Провайдер | Статус Docker / DB | Доступные типы БД | Главные минусы платформы | Критично для категорий |
| :--- | :---: | :--- | :--- | :---: |
| **Oracle Cloud** | **БЕСПЛАТНО** | Любые внутри Docker (PostgreSQL, SQLite, MySQL) | **Жесткий KYC.** Требует привязки карты. Частые баны аккаунтов без объяснения причин. | **Все категории** (Сложно пройти регистрацию) |
| **Hugging Face** | **ПЛАТНО** | Только внешние базы данных через сторонние API | **Docker платный ($9/mo).** Диск не персистентный — при рестарте контейнера локальная БД сотрется. | **A, B, C** (Только для мелких скриптов) |
| **Render / Railway** | **ПЛАТНО** | PostgreSQL, Redis, MySQL | Бесплатные лимиты быстро тают. Платные базы стоят дорого при росте объемов информации. | **B, C, D** (Дорого при масштабировании) |
| **Vercel / GitHub Pages**| **БЕСПЛАТНО** | Только внешние Serverless БД (Supabase, Neon) | **Нет поддержки Docker.** Невозможно запустить ядро TDLib. Подходит только для промо-страниц. | **D, E, F, G** (Бесполезно для бэкенда) |
| **AnubizHost** | **ПЛАТНО** | Любые DB внутри выделенного Docker-контейнера | Высокая базовая стоимость аренды. Медленная техподдержка в тикетах. | **A, B** (Невыгодно для базовых тестов) |
| **Coin.Host** | **ПЛАТНО** | Идеально для тяжелых кластеров баз данных | **Обязательная проверка номера телефона.** Жесткий внутренний мониторинг за расходом CPU. | **F, G** (Нужна точечная настройка лимитов) |
| **Space-Core** | **ПЛАТНО** | Любые DB на скоростных массивах NVMe | Иногда запрашивают текстовое описание проекта перед выдачей серверов с портом 10 Gbps. | **E, F, G** (Критично при первом заказе) |
| **Hetzner / OVH** | **ПЛАТНО** | Любые типы БД (Огромные RAID-массивы) | **Тотальный KYC по паспорту. Нулевая абузоустойчивость:** сервер удалят при первых жалобах на неофициальный API. | **D, E, F, G** (Высокий риск потерять все данные) |
